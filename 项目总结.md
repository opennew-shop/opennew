## 1. 项目概述

ANCF (Agent-Native Commerce Framework) 是一个**零固定前端**的商业协议：平台不提供传统重型商城前端，而是由 Agent 发现 manifest、拉取商品 JSON、按固定固件/模板在本地临时渲染交互界面，再把用户意图交回后端完成 quote、checkout、支付、账本与服务开通。

核心信任边界：

- 本地 HTML / Vue / Web Components 只负责展示与意图采集，**不是可信交易源**。
- search 返回的价格、库存、文案都不可作为成交依据。
- checkout 必须经过后端 quote → checkout_prepare → 钱包签名 → checkout_commit。
- 账本、铸币、支付、服务开通一律以后端状态机为准。

## 4. 分层架构与代码组织约定

每个 Go 服务遵循统一的整洁分层（cmd 入口 + internal 四层）：

```text
cmd/main.go            # 进程入口：装配依赖、注册路由、启动 HTTP
internal/handler/      # 接入层：解析请求、调用 service、组织响应
internal/service/      # 业务逻辑层：状态机、事务编排、领域规则
internal/repository/   # 数据访问层：SQL、行锁、Outbox 写入
internal/model/        # 领域模型与 DTO（struct + db/json tag）
internal/middleware/   # （部分服务）鉴权等横切关注点
```

依赖方向严格单向：`handler → service → repository → model`，事务边界由 service 层持有（典型如 ledger 的 mutation 方法要求调用方传入 `*sql.Tx`，保证账本写入与其他领域操作的原子性）。

## 5. 关键设计决策

1. **影子账本 AGP（原 vUSDC）**：MVP 阶段为内部账本，不在链上铸币，恒等约束 `total_liability + pending_redemption ≤ confirmed_reserve`。
2. **双分录不可变账本**：余额不直接改写，所有变更落为 immutable ledger entries（借/贷成对）。
3. **Outbox 模式**：数据库事务内不调用外部服务，事件先入 outbox 表，再由处理器异步发布，保证跨系统最终一致性。
4. **并发与隔离**：Phase 5 由 `SERIALIZABLE` 调整为 `READ COMMITTED` + `FOR UPDATE SKIP LOCKED` + CASE-WHEN 批量扣减，兼顾正确性与吞吐。
5. **幂等三路解析**：checkout commit 按幂等键区分“回放 / body-hash 冲突 / 新请求”。
6. **零固定前端**：展示层（HTML/Vue/WC）不可信，成交必须回到后端 quote→prepare→签名→commit。
7. **链上演进**：Phase 3 影子账本 → Phase 4 可选 SPL Token-2022（2-of-3 多签）+ 链上 supply 对账。

## 6. 安全措施（代码层已落地）

API Key 鉴权（constant-time 比较）、Redis 令牌桶限流、JSON Schema 校验、HTTP 签名（RFC 9421）、EdDSA 钱包签名、幂等键唯一约束、报价短 TTL + 原子消费、库存并发锁、双分录不可变账本、pg advisory lock 防双花、Outbox、CSP 头（禁 eval）、固件 SRI(sha384)、不可变审计日志、储备对账、风控（日限额/每钱包限额/大额审批）、服务间内部鉴权（X-Internal-API-Key）、2-of-3 多签 mint authority。

配置安全：`config.Load()` 不再提供 `JWT_SECRET`/`API_KEY` 的硬编码默认值，缺失即 `log.Fatalf` 启动失败（见 [config.go](services/api-gateway/internal/config/config.go)）。
